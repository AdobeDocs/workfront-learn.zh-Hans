# 最佳实践 — 组织单位

## 什么是Adobe Workfront“最佳实践”？

最佳实践是指导准则，它们代表了一个有效、高效的操作过程；您和您公司的用户可轻松采用；并且可在您的整个组织中成功复制。

在查看这些建议时，请牢记，某些Workfront最佳实践是通用的，而其他最佳实践可能更特定于此主题。 使用这些最佳实践作为框架，帮助指导您的Workfront系统设置和使用。

## 导航此页面

滚动浏览本页时，首先您会找到该主题所有最佳实践的高级列表。 这样，您就可以查看推荐，而无需深入了解“原因”的详细信息。

“为什么这些是最佳实践？” 区域位于高级列表之后，它提供了一些最佳实践的更多详细信息，以及为什么将它们视为流程、工具等，您应考虑使用Workfront实例进行实施。

</br>
</br>

## 组织单位最佳实践

* 在Workfront中设置组织单位之前，确定关键报表类别和需求。
* 定期评估您现有的Workfront组织结构，确保它满足您的需求。
* 每季度或至少每六个月审核一次您的用户群。

* 识别并培训用户成为组管理员。

* 向用户传达什么是Workfront的良好状态、Workfront功能有什么变化，或者如何更新流程/工作流。

* 您的内部用户通常只需要一家公司。

* 为要在Workfront实例中跟踪或需要登录实例的外部客户或供应商创建公司。

* 最大限度地减少Workfront组，并将其与贵组织的部门相关联。

* 使用组表示部门结构，使用团队表示工作结构。
* 利用组管理员帮助系统管理员管理用户和设置。

* 允许组管理员创建自己的子组。

* 将布局模板分配给主团队。

* 用户的主团队应该是他们最常合作的团队。

* 使用较大的组织单位与人员集共享项目。

</br>

## 为什么有这些最佳实践？

**最佳实践**

在Workfront中设置组织单位之前，确定关键报表类别和需求。



**原因如下**

通过了解您需要什么量度，以及希望在Workfront中收集哪些有关用户及其所做工作的信息，您可以专注于构建组织单位以满足这些需求。

</br>
</br>

**最佳实践**

定期评估您现有的Workfront组织结构，确保它满足您的需求。



**原因如下**

随着贵企业的发展或Workfront在贵企业内的扩展，您将找到新的和不同的方法来提高流程效率以及用户与工作的关系。 因此，定期评估组织单位的构建方式可以让您有机会知道该结构是否仍然有效或是否需要更新。

</br>
</br>


**最佳实践**

每季度或至少每六个月审核一次您的用户群。



**原因如下**

随着您的组织逐渐使用Workfront并增加该工具的使用量，信息就会在系统中累积。 这包括用户数量、使用的许可证数量、已创建的公司、组或团队数量等。



随着时间的推移，您的组织也可能会经历一些更改。 例如，也许上个季度发生了部门重组。 或者，一些员工决定寻找其他机会。 这意味着您让Workfront中的用户接受许可证，并且团队或组中的差距可能会影响资源估计或工作分配。



对Workfront中的组织单位和用户进行季度审核是保持系统顺畅高效运行的关键。 这也是确保Workfront利用率稳定或增长以及员工使用Workfront进行日常工作管理（而不是电子邮件或电子表格）的大好时机。

</br>
</br>

**最佳实践**

识别并培训用户成为组管理员。



**原因如下**

选择对Adobe Workfront及其团队如何使用系统有充分了解的人员，以及有时间专门担任组管理员的人员。



组管理员可以为他们管理的组和子组设置状态、修改项目偏好设置或创建时间表配置文件。 他们还可以管理群组的用户、调整访问级别和分配布局模板。



这可以将系统管理员从Workfront的一些日常设置和维护中解放出来。 此外，它还使每个组能够更好地控制自己的设置和操作首选项，而不会妨碍Workfront中的其他组。

</br>
</br>


**最佳实践**

您的内部用户通常只需要一家公司。



**原因如下**

一家Workfront公司应该服务于整个组织，而不仅仅是业务领域。 如果您将内部用户拆分为不同的公司，则可能会导致在Workfront中工作的权限出现问题。 Workfront组是组织内部用户的首选方法。

</br>
</br>

**最佳实践**

为要在Workfront实例中跟踪或需要登录实例的外部客户或供应商创建公司。



**原因如下**

通过为外部供应商创建公司，您可以允许Workfront中的这些供应商审查为其构建的内容或产品的进度，同时限制他们在系统中的访问权限。



授予Workfront访问权限时，将组合分配对象共享和访问级别给供应商公司中的用户。 例如，您可以设置限制，以便这些用户只能看到自己公司中的人员，和/或向特定项目（如请求队列和请求）授予权限，以便只有特定公司的用户才能看到这些项目。 这有助于确立您在Workfront中保留信息的隐私。

</br>
</br>


**最佳实践**

最大限度地减少Workfront组，并将其与贵组织的部门相关联。



**原因如下**

在Workfront中管理工作的部门大多需要小组。 例如，营销部门可能需要自己的组，但来自其他部门、仅在Workfront中请求工作的用户可放入默认组。

</br>
</br>

**最佳实践**

使用组表示部门结构，使用团队表示工作结构。



**原因如下**

主组是用户和项目的必填字段。 这样可以很好地组织和报告用户及其参与的高层次工作。



但是，当涉及到向一组人员执行任务（通过项目中的任务或通过请求队列）时，只能向团队执行任务。

</br>
</br>

**最佳实践**

利用组管理员帮助系统管理员管理用户和设置。



**原因如下**

系统管理员有权访问Workfront中的所有内容，包括全局系统设置。 系统管理员可以访问的设置组由系统管理员控制，并且仅应用于该特定组。



拥有组管理员让系统管理员可以委派许多职责，让他们专注于更大的项目，而不是Workfront的日常维护。 组管理员可以更轻松地与组的需求保持联系，从而为用户提供更好的服务。

</br>
</br>


**最佳实践**

允许组管理员创建自己的子组。



**原因如下**

组管理员可以创建和管理自己的子组。 虽然系统管理员可以创建子组，但将子组委派给组管理员可能非常节省时间。 这还可以为群组成员带来更好的Workfront体验，因为群组管理员将更熟悉群组和子组的需求和工作流。

</br>
</br>

**最佳实践**

将布局模板分配给主团队。



**原因如下**

确保您的团队在工作时处于同一页面非常重要。 这意味着让他们更轻松地访问相同的报告和工具。 布局模板允许您创建该环境，并增强甚至可能改进团队已制定的流程。

</br>
</br>

**最佳实践**

使用较大的组织单位与人员集共享项目。



**原因如下**

在Workfront中共享项目、报告等项时，使用较大的组织单位，如组和团队。 — 意味着您可以快速轻松地与一组选定的人员同时共享对象。 将个人用户的名称添加到共享窗口可能会很麻烦且耗时。



这样还可以更轻松地维护项目上的共享。 随着团队成员在Workfront中更新，项目的共享权限会自动更新，从而授予新团队成员对项目的访问权限，并删除前团队成员的访问权限。

# 最佳实践 — 组织单位

## 什么是Adobe Workfront“最佳实践”？

最佳做法是代表有效、高效的行动方针的准则；易于您和贵公司用户采用；和可以在您的组织内成功复制。

在您查看这些建议时，请记住，一些Workfront最佳实践是普遍的，而其他一些则可能更具体地介绍该主题。 将这些最佳实践用作框架，以帮助指导Workfront系统的设置和使用。

## 导航此页面

在您浏览此页面时，首先您会找到该主题所有最佳实践的高级列表。 这样，您便可以审核推荐，而无需深入研究“原因”的详细信息。

“为什么这些最佳实践？” 高级列表后找到的区域，详细介绍了一些最佳实践以及它们被视为流程、工具等的原因，您应当考虑使用Workfront实例进行实施。

</br>
</br>

## 组织单位最佳实践

* 在Workfront中设置组织单位之前，确定关键报告类别和需求。
* 定期评估您现有的Workfront组织结构，以确保其满足您的需求。
* 每季度审核您的用户群，或至少每六个月审核一次。

* 识别和培训用户以成为群组管理员。

* 向您的用户传达Workfront的运行状况、Workfront功能即将发生的更改，或有关流程/工作流更新的信息。

* 内部用户通常只需要一个公司。

* 为要在Workfront实例中跟踪或需要实例登录权限的外部客户或供应商创建公司。

* 将Workfront组保持在最低值，并将它们与贵组织的部门关联。

* 使用组来表示部门结构，使用组来表示工作结构。
* 利用组管理员帮助系统管理员管理用户和设置。

* 允许组管理员创建自己的子组。

* 将布局模板分配给主团队。

* 用户的主团队应该是他们最常与之合作的团队。

* 使用较大的组织单位与人员集共享项目。

</br>
</br>


## 为什么是这些最佳实践？

**最佳实践**

在Workfront中设置组织单位之前，确定关键报告类别和需求。



**原因如下**

了解您要查找的量度以及要在Workfront中收集的有关用户及其工作的信息，使您能够专注于构建组织单位以满足这些需求。

</br>
</br>

**最佳实践**

定期评估您现有的Workfront组织结构，以确保其满足您的需求。



**原因如下**

随着您的组织的发展，或随着Workfront在您的组织内部的扩展，您将找到新的、不同的方法来提高流程效率以及用户与工作的关系。 因此，定期评估组织单位的结构方式可以让您有机会了解该结构是否仍然有效或是否需要更新。

</br>
</br>


**最佳实践**

每季度审核您的用户群，或至少每六个月审核一次。



**原因如下**

随着您的组织逐渐使用Workfront并增加了工具的使用，信息会在系统中构建。 这包括用户数量、使用的许可证数量、已创建的公司、组或团队的数量等。



随着时间的推移，您的组织可能也会经历更改。 例如，上个季度可能发生了部门重组。 或许有些员工决定去看其他机会。 这意味着您的Workfront用户正在使用许可证，并且团队或组中可能存在可能影响资源估算或工作分配的差距。



对Workfront中的组织单位和用户进行季度审计是保持系统平稳高效运行的关键。 现在也是确保Workfront利用率稳定或增长、员工使用Workfront进行持续工作管理（而非电子邮件或电子表格）的好时机。

</br>
</br>

**最佳实践**

识别和培训用户以成为群组管理员。



**原因如下**

选择对Adobe Workfront及其组如何使用系统有深入了解的人员，以及有时间专门担任组管理员的人员。



组管理员可以设置状态、修改项目首选项，或为他们管理的组和子组创建时间表配置文件。 他们还可以管理其组的用户、调整访问级别和分配布局模板。



这样，系统管理员就不必再进行某些日常设置和维护Workfront。 它还允许每个组更好地控制自己的设置和操作首选项，而不会干扰Workfront中的其他组。

</br>
</br>


**最佳实践**

内部用户通常只需要一个公司。



**原因如下**

一家Workfront公司应该面向整个组织，而不仅仅是业务领域。 如果将内部用户拆分为不同的公司，则可能会导致在Workfront中工作的权限出现问题。 Workfront组是组织内部用户的首选方法。

</br>
</br>

**最佳实践**

为要在Workfront实例中跟踪或需要实例登录权限的外部客户或供应商创建公司。



**原因如下**

通过与外部供应商创建公司，您可以允许这些供应商进入Workfront，在限制他们在系统中访问的同时，检查为他们构建的内容或产品的进度。



Workfront访问权限是通过分配给供应商公司中用户的对象共享和访问级别的组合授予的。 例如，您可以设置限制，以便这些用户只能查看其自己公司中的人员和/或授予对某些项目（例如请求队列和请求）的权限，以便只有特定公司的用户才能看到这些项目。 这有助于确保您在Workfront中保留的信息的隐私。

</br>
</br>


**最佳实践**

将Workfront组保持在最低值，并将它们与贵组织的部门关联。



**原因如下**

在Workfront管理工作的部门大多需要小组。 例如，营销部门可能需要其自己的组，但是来自其他部门(只在Workfront中)的请求工作的用户可以被放置在默认组中。

</br>
</br>

**最佳实践**

使用组来表示部门结构，使用组来表示工作结构。



**原因如下**

对于用户和项目，“主页”组是必填字段。 这为组织和报告用户及其参与的高级工作提供了一种良好的方法。



但是，在对一组人员进行分配（通过项目上的任务或通过请求队列）时，只能对团队进行分配。

</br>
</br>

**最佳实践**

利用组管理员帮助系统管理员管理用户和设置。



**原因如下**

系统管理员有权访问Workfront中的所有内容，包括全局系统设置。 设置组管理员可以访问的权限由系统管理员控制，并且只能应用于该特定组。



通过让组管理员，系统管理员可以委派许多责任，从而让他们能够专注于大图景项目，而不是日常维护Workfront。 群组管理员可以更轻松地与群组的需求保持联系，从而为用户提供更好的服务。

</br>
</br>


**最佳实践**

允许组管理员创建自己的子组。



**原因如下**

群组管理员可以创建和管理自己的子群组。 虽然系统管理员可以创建子组，但将此子组委派给组管理员可以节省大量时间。 此外，它还可以为群组成员提供更好的Workfront体验，因为群组管理员将更加熟悉群组和子组的需求和工作流。

</br>
</br>

**最佳实践**

将布局模板分配给主团队。



**原因如下**

确保团队在工作时位于同一页面上，这一点非常重要。 这意味着他们能够更轻松地访问相同的报表和工具。 通过布局模板，您可以创建该环境，并增强团队已有的流程（可能会改进）。

</br>
</br>

**最佳实践**

使用较大的组织单位与人员集共享项目。



**原因如下**

在Workfront中共享项目、报表等时，使用组和团队等更大的组织单位。 — 表示您可以同时快速轻松地与一组选定的人员共享对象。 在共享窗口中添加单个用户的名称可能既繁琐又耗时。



这也更便于维护项目上的共享。 当团队成员在Workfront中更新时，项目的共享权限会自动更新，从而授予新团队成员对该项目的访问权限并移除前团队成员的访问权限。
